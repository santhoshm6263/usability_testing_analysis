Usability test plan
A test plan defines exactly who, what, when, and how you test — written before any session begins. Think of it as your project brief for testing.
Product being tested
TaskFlow — a mobile task management app (Figma clickable prototype). The prototype covers 6 screens: Splash, Onboarding, Login, Home, Add Task, and Profile.
Testing objective
Identify usability issues that prevent users from completing core tasks efficiently. Measure task completion rate, time-on-task, and satisfaction. Validate or invalidate key design decisions before development begins.
Participant criteria
Age 18–35
Smartphone user daily
Has used any to-do app
Not a UX designer
Session structure
Pre-session briefing (5 min)
Explain the process, get consent, clarify this tests the app — not the user.
Think-aloud warm-up (3 min)
Ask user to narrate actions on a simple unrelated task to get comfortable.
3 test scenarios (15 min)
One scenario at a time. Observer watches and notes without helping.
Post-session interview (7 min)
5 questions about experience. SUS questionnaire completed by participant.
What the observer records
For each task: did they complete it (Y/N), how long it took (seconds), any hesitations or wrong taps, and any verbal comments. These go into the observation log (see Observations tab).
Test scenarios
Each scenario gives the user a realistic goal — never a UI instruction. "Tap the blue button" tells them what to do; "find out how to add a task" tests whether the design communicates it.
Scenario 1
First launch to home screen
Goal: Complete onboarding and reach the Home screen unaided. Measures: time to home, skip rate, login confusion.
Instructions read to participant:
"Imagine you've just downloaded this app. Open it and get yourself set up with an account. Tell me when you feel you've arrived at the main screen."
Target: under 90 sec Screen: Splash → Home
Scenario 2
Add a new task with details
Goal: Create a task with a title, due date (today), and priority colour. Measures: FAB discoverability, form completion rate, time-on-task.
Instructions read to participant:
"You're on the main screen. Add a new task called 'Buy groceries', set it for today, and mark it as high priority. Save it."
Target: under 45 sec Screen: Home → Add Task
Scenario 3
Complete a task and check stats
Goal: Mark a task complete and navigate to Profile to view completion stats. Measures: checkbox discoverability, nav bar usability, stat comprehension.
Instructions read to participant:
"Mark the 'Buy groceries' task as done. Then find out how many tasks you've completed in total since you started using the app."
Target: under 30 sec Screen: Home → Profile

Findings & analysis
Each finding is a named, severity-rated usability issue backed by tester evidence. This is the core of your report — recruiters and reviewers read this section most carefully.
Finding 1 — Priority colour ambiguity
High severity
4 of 5 testers could not correctly interpret the priority colour dots. Users spent 15–22 extra seconds on this element, and 2 testers failed to set priority at all.
"I don't know what these mean" — Kavya  |  "Are these decoration?" — Sneha
Root cause: Colour alone carries no label. Users have no shared mental model for red = urgent unless told.
Finding 2 — FAB not discovered promptly
High severity
3 of 5 testers took longer than 15 seconds to find the + button. 1 tester (Sneha) never found it and required moderator intervention.
"Where do I add?" — Priya (after 38 seconds)
Root cause: FAB blends into the screen. No label, no first-use highlight, and 48px size is too small for users unfamiliar with the pattern.
Finding 3 — Profile icon hard to locate
Medium severity
4 of 5 testers tapped a wrong bottom nav icon before finding Profile. Average extra time: 12 seconds. Testers expected Profile at far right of nav.
Root cause: Profile icon is 3rd of 4 icons, not last. Convention (iOS, Android) places Profile/Account as the rightmost nav item.
Finding 4 — Stats labels are ambiguous
Medium severity
2 of 5 testers were unsure whether "Tasks completed" meant all-time or today. Caused confusion on Scenario 3.
"Does that mean completed today, or ever?" — Sneha
Root cause: Stat labels lack a time scope. "23 tasks" with no qualifier is ambiguous.
Positive finding — Task completion animation
5 of 5 testers expressed verbal delight at the checkmark spring animation. "I like that" (Arjun), "Oh nice!" (Kavya). This is a core differentiator — retain and amplify.
Design improvements
Each improvement maps directly to a finding. This is what your v2 prototype should change — and what you test in the next round.
Fix 1 — Replace colour dots with labelled priority tags
Change the 4 colour circles to pill-shaped tags: "Low", "Medium", "High", "Urgent" — each with its colour as background. Label removes all ambiguity. No cognitive effort needed.
Fixes Finding 1Effort: low
Fix 2 — Enlarge FAB + add first-use label
Increase FAB from 48px to 58px. On first launch only, show a brief label "Add task" beside the button (fades after 3 seconds). Add a one-time pulse ring animation to draw attention.
Fixes Finding 2Effort: medium
Fix 3 — Move Profile icon to rightmost nav position
Reorder bottom nav: Home · Search · Calendar · Profile (right). Follows iOS and Android conventions that 100% of testers expected. One icon move, zero design work.
Fixes Finding 3Effort: minimal
Fix 4 — Add time scope to stat labels
Change "Tasks completed" to "Tasks completed (all time)". Add a secondary line: "7 completed today". Two lines, zero ambiguity.
Fixes Finding 4Effort: minimal

Full report summary
Case study format: Problem → Process → Findings → Solution → Outcome. Copy this into your GitHub README or portfolio submission.
Problem
The TaskFlow prototype was designed to simplify task management. Before development, we needed to validate whether users could complete the 3 core tasks — onboarding, adding a task, and tracking progress — without confusion or errors.
Process
Recruited 5 participants matching the target persona (ages 19–34, daily smartphone users). Conducted 30-minute moderated usability sessions using a Figma clickable prototype. Each session covered 3 scenarios using the think-aloud method. Collected both quantitative data (task time, completion rate) and qualitative data (verbal comments, hesitations).
Key findings
4 usability issues identified: (1) Priority colour dots had no labels — 4/5 testers confused. (2) FAB button not discovered by 3/5 testers quickly. (3) Profile nav icon in wrong position for platform conventions. (4) Stats labels had no time scope. One strong positive: task completion animation received universal delight (5/5). Overall SUS score: 67 — borderline acceptable.
Solution
Four targeted design changes proposed for v2: labelled priority tags, larger FAB with first-use tooltip, Profile moved to rightmost nav position, and stat labels with time scope added. All changes are low-to-medium effort and do not require new screens — only component-level edits.
Projected outcome
Target for v2 round: Task 2 completion rate from 40% to 80%+. Average time on Scenario 2 from 68 sec to under 45 sec. SUS score from 67 to 75+. Two new testers and one returning tester will validate the changes before development begins.
